GET은 불필요한 요청을 제한하기 위해 요청이 캐시될 수 있다. 데이터의크기가 크고 변경될 일이 적은 것들은 반복해서 동일한 요청을 보낼 필요가 없기 때문에 정적컨텐츠를 요청하고 나면 브라우저에서는 요청을 캐시해두고 캐시된 데이터를 사용 그래서 프론트엔드 개발을 하다보면 정적컨텐츠가 캐시돼 컨턴츠를 변경해도 내용이 바뀌지않아 강제 리프래쉬를 시켜 줍니다. 이때는 브라우저의 캐시를 지워주면 다시 컨텐츠 조회를 하기위해 서버로 요청을 보냅니다 POST는 리소스를 생성/변경하기 위해 설계 되었기 때문에 GET과 달리 전송해야될 데이터를 HTTP 메세지의 BODY에 담아서 전송합니다. HTTP메세지의 BODY는 길이의 제한이 없이 데이터를 전송할 수 있다. 따라서 POST요청은 GET요청과 달리 대용량의 데이터를 전송할 수 있다 POST는 데이터가 바디를 통해 전송되고 내용이 눈에 보이지 않아 안전하다고 생각 할 수 있지만 POST요청도 크롬 개발자도구 Fiddler와 같은 툴로 요청 내용을 확인 할 수 있기 때문에 민감한 데이터의 경우 반드시 암호화해 전송해야 합니다. POST요청을 보낼 때 요청 헤더의 Content-Type에 요청 데이터 타입을 표시해야 합니다. 데이터 타입을 표시 하지 않으면 서버는 내용이나 URL에 포함된 리소스의 확장자명 등으로 데이터 타입을 유추합니다. GET은 서버에게 동일한 요청을 여러번 전송하더라도 동일한 응답이 돌아와야합니다. Post는 서버에게 동일한 요청을 여러번 전송해도 응답은 항상 다를 수 있습니다. 따라서 POST는 서버의 상태나 데이터를 변경시킬 때 사용합니다. 게시글을 쓰면 서버에 게시글이 저장되고 게시글을 삭제하면 해당데이터가 없어지는 등 Post로 요청을 하게되면 서버의 무언가는 변경되도록 사용됩니다. 하지만 생성에는 POST, 수정은 PUT 또는 PATCH, 삭제는 DELETE가 더 용도에 맞는 메소드 입니다. RSET이란 자원을 이름으로 구분하여 해당 자원의 상태를 주고받는 모든 것을 의미합니다. Json 혹은 XML를 통해 데이터를 주고 받는 것이 일반적입니다. REST은 기본적으로 웹의 기존 기술과 HTTP프로토콜을 그대로 활용하기 때문에 웹의 장점을 최대한으로 활용할 수 있는 아키텍처 스타일 입니다. REST는 네트웨크 상에서 Clint와 Server 사이의 통신방식중 하나입니다. 구체적인 개념으로 HTTP URL을 통해 자원을 얻고 Method(get,post, put, delete)를 통해 자원에 대한 CRUD opreation을 적용하는 것을 의미합니다. 웹사이트의 이미지, 텍스트 DB내용등의 모든 자원에 고유한 ID인 HTTP URL을 부여합니다. HTTP 프로토콜은 TCP/IP 3계층에서 작동하는 HTML,이미지 동영상등 종류를 보낼 수 있게 하는 서버/클라이언트 모델을 따르는 약속같은 느낌 클라(request) -> 서버(response) 보통 80번 포트 사용 Cookie는 클라이언트, 서버의 상태 정보를 담고있는 정보 조각 200번때 성공 , 400번때 실패 CRUD Operation create, read(조회), Update, delete, HEAD(header정보조회) HTTP프로토콜 인프라를 그대로 사용하므로 REST API사용을 위한 별도의 인프라 구축이필요없음 HTTP표준 프로토콜을 따르는 모든 플랫폼에서 사용가능, 서버 클라이언트 역할 명확히 분리 BUT, 사용할 수 있는 메소드가 제한적 REST이 필요한 이유는 다양한 클라이언트들의 등장, 멀티 플랫폼에 대한 지원을 위해 서비스 자원에 대한 아키텍쳐를 세우고 이용하려고 REST의 구성요소로 자원, 행위, 표현이 있다. REST의 특징으로는 서버, 클라이언트 구조 Stateless(무상태), Cacheable(캐시 처리가능), 계층화 인터페이스의 일관성 등이 있다.
