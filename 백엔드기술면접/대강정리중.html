GET은 불필요한 요청을 제한하기 위해 요청이 캐시될 수 있다. 데이터의크기가 크고 변경될 일이 적은 것들은 반복해서 동일한 요청을 보낼 필요가 없기 때문에 정적컨텐츠를 요청하고 나면 브라우저에서는 요청을 캐시해두고 캐시된 데이터를 사용 그래서 프론트엔드 개발을 하다보면 정적컨텐츠가 캐시돼 컨턴츠를 변경해도 내용이 바뀌지않아 강제 리프래쉬를 시켜 줍니다. 이때는 브라우저의 캐시를 지워주면 다시 컨텐츠 조회를 하기위해 서버로 요청을 보냅니다 POST는 리소스를 생성/변경하기 위해 설계 되었기 때문에 GET과 달리 전송해야될 데이터를 HTTP 메세지의 BODY에 담아서 전송합니다. HTTP메세지의 BODY는 길이의 제한이 없이 데이터를 전송할 수 있다. 따라서 POST요청은 GET요청과 달리 대용량의 데이터를 전송할 수 있다 POST는 데이터가 바디를 통해 전송되고 내용이 눈에 보이지 않아 안전하다고 생각 할 수 있지만 POST요청도 크롬 개발자도구 Fiddler와 같은 툴로 요청 내용을 확인 할 수 있기 때문에 민감한 데이터의 경우 반드시 암호화해 전송해야 합니다. POST요청을 보낼 때 요청 헤더의 Content-Type에 요청 데이터 타입을 표시해야 합니다. 데이터 타입을 표시 하지 않으면 서버는 내용이나 URL에 포함된 리소스의 확장자명 등으로 데이터 타입을 유추합니다. GET은 서버에게 동일한 요청을 여러번 전송하더라도 동일한 응답이 돌아와야합니다. Post는 서버에게 동일한 요청을 여러번 전송해도 응답은 항상 다를 수 있습니다. 따라서 POST는 서버의 상태나 데이터를 변경시킬 때 사용합니다. 게시글을 쓰면 서버에 게시글이 저장되고 게시글을 삭제하면 해당데이터가 없어지는 등 Post로 요청을 하게되면 서버의 무언가는 변경되도록 사용됩니다. 하지만 생성에는 POST, 수정은 PUT 또는 PATCH, 삭제는 DELETE가 더 용도에 맞는 메소드 입니다.
